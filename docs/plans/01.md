# 二维码图片检测首版方案

  ## Summary

  目标是在管理平台图片上传链路中，服务端同步检测图片是否包含二维码样式区域，用于高召回拦截，并提示人工处理。首版采用单类别目标检测：只判断是否存在
  qrcode，不区分常规码、艺术码、小程序码、抖音码等子类型。

  核心策略：

  - 真实历史上传图作为主训练数据。
  - 合成二维码贴图只作为数据增强补充，不作为主数据集。
  - 全图检测，但在训练采样和推理后处理上对左下/右下区域做适度先验加权。
  - 选择轻量检测模型，保证服务端单张 1 秒内可落地。

  ## Key Changes

  ### 数据与标注

  - 从历史上传图片中导出首批数据，优先覆盖：
      - 含码图：常规码、艺术码、小程序码、抖音码、低对比度码、部分遮挡码、边缘位置码。
      - 不含码图：演出海报、宣传图、票务图、人物图、文字密集图、Logo/装饰图，尤其要包含“像码但不是码”的干扰样本。
  - 标签统一为一个类别：qrcode。
  - 标注规则固定为：
      - 只框出二维码可见外接矩形。
      - 一个图片多个码时分别标注。
      - 码被裁切或部分遮挡时，仍按可见区域对应的完整码外接框近似标注。
      - 不标注普通文字块、Logo、舞台灯牌等非二维码元素。
  - 数据集按 train/val/test 拆分，保证正负样本和不同码样式在各集合中分布稳定。

  ### 训练方案

  - 使用轻量 YOLO 检测模型的 nano/small 级别作为首版基线。
  - 输入尺寸先以 960 或 1024 为基线实验，优先保证小码召回，不先追求极限速度。
  - 数据增强以真实感优先：
      - 缩放、轻透视、亮度/对比度、模糊、压缩噪声、局部遮挡。
      - 少量 Mosaic/Copy-Paste 可保留，但强度要低，避免偏离真实业务图片分布。
  - 合成数据生成策略只作为补充：
      - 从公开或自生成二维码样式库中准备不同视觉风格的码图。
      - 将码贴到真实“无码”演出图片的左下/右下为主，也保留少量非角区位置。
      - 贴图时加入缩放、旋转、透视、透明度、边缘融合、噪声和压缩。
      - 合成样本占比控制在较低水平，避免模型过拟合合成痕迹。
  - 评估指标以高召回优先：
      - 主看图片级召回率、目标级召回率、误报率。
      - 次看 mAP。
      - 阈值选择以“拦截并提示人工处理”场景为准，不按通用检测榜单调参。

  ### 推理与服务集成

  - 服务端上传后同步执行检测。
  - 推理逻辑：
      - 对原图按长边缩放到模型输入尺寸。
      - 执行单类别检测。
      - 对左下/右下区域内的候选框给予轻度分数加权，但不屏蔽其他区域。
      - 若任一候选框分数超过业务阈值，则判定为“疑似含二维码”。
  - 对外接口保持简单：
      - 输入：图片文件或图片 URL。
      - 输出：has_qrcode: bool、score: float、boxes: [[x1,y1,x2,y2,score]]。
  - 业务动作：
      - 命中后拦截上传或阻断提交，并提示人工处理。
      - 同时记录检测分数、框位置、图片标识，供后续误报/漏报回流。

  ## Test Plan
  - 数据集层面：
      - 单独输出角区样本与非角区样本表现，确认先验加权没有造成明显偏置漏检。
  - 服务层面：
      - 单张 1k~2k 图片同步检测耗时满足目标。
      - 上传失败/模型异常时有兜底策略和日志。
      - 返回框坐标可用于后台人工复核。

  ## Assumptions
  - 首版只保留一个类别 qrcode，不区分码类型。
  - 历史上传图可以导出上千张，并允许进行人工标注。
  - 允许通过人工复核承接一部分误报，以换取更高召回。
  - iMac 无 GPU 条件下，首版训练以轻量模型和中小规模数据集为主；如果后续需要更高上限，再迁移到云 GPU 做系统性调参。
